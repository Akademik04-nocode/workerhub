# WorkerHub

Telegram Mini App для быстрого подбора работников и поиска подработок.
Монорепозиторий: `backend` (Fastify + Drizzle + PostgreSQL + Redis), `bot` (grammY), `frontend` (React + Vite).

## Установка на сервер (для неспециалиста)

Нужны: сервер Ubuntu 24.04 (2 CPU / 2 ГБ), домен с A-записью на IP сервера,
токен бота от @BotFather и ваш Telegram ID (узнать: @userinfobot).

```bash
git clone <репозиторий> /opt/worker-hub
cd /opt/worker-hub
sudo bash install.sh
```

Скрипт задаст три вопроса (домен, токен, ID), сам поставит Docker, настроит
файрвол, сгенерирует пароли, запустит приложение и получит HTTPS-сертификат
(Caddy, продлевается автоматически). Останется привязать кнопку в @BotFather:
Bot Settings → Menu Button → адрес `https://ваш-домен`.

## Обновление командой /obnovit (без консоли)

Чтобы обновлять приложение прямо из Telegram, один раз включите фоновый
обновлятор на сервере (после того, как код с командой уже развёрнут):

```bash
chmod +x /opt/worker-hub/deploy-watcher.sh
( crontab -l 2>/dev/null | grep -v deploy-watcher; \
  echo "* * * * * /usr/bin/flock -n /tmp/wh-deploy.lock /opt/worker-hub/deploy-watcher.sh" ) | crontab -
```

После этого администратор (Telegram ID из `ADMIN_TELEGRAM_IDS`) отправляет боту
`/obnovit` — бот ставит флаг в Redis, `deploy-watcher.sh` по крону (раз в минуту)
видит его, выполняет `git pull` и пересборку `backend/frontend/bot`, а затем
присылает в Telegram «✅ Обновление завершено». Бот не имеет прямого доступа к
Docker или файлам сервера — только ставит флаг. Лог обновлений: `deploy.log`.

## Быстрый старт вручную

```bash
cp .env.example .env   # заполните DOMAIN, BOT_TOKEN, POSTGRES_PASSWORD, ADMIN_TELEGRAM_IDS
docker compose up -d --build
```

Миграции применяются автоматически при старте backend. Весь внешний трафик
идёт через Caddy (порты 80/443): статика фронтенда и `/api/*` на одном домене;
порты Postgres, Redis, backend и frontend наружу не публикуются.

Локально без Docker:

```bash
# backend (нужны запущенные PostgreSQL и Redis, см. .env)
cd backend && npm ci && npm run dev
# frontend
cd frontend && npm ci && npm run dev
# bot
cd bot && npm ci && npm run dev
```

Новые миграции после изменения `src/db/schema.ts`: `npm run db:generate`
(папка `drizzle/`, включая `drizzle/meta/`, коммитится в репозиторий).

## Безопасность и конфигурация

- **Администраторы** задаются только через `ADMIN_TELEGRAM_IDS` (пусто = ни одного
  бутстрап-админа); дальше роли назначаются через админ-панель.
- **PostgreSQL и Redis не публикуют порты наружу** — доступны только внутри сети
  compose. `POSTGRES_PASSWORD` обязателен, compose не поднимется с пустым значением.
- **CORS**: пустой `CORS_ORIGIN` разрешает все origin (только для dev); в проде
  укажите домен мини-приложения.
- **Rate limiting**: 120 запросов/мин на пользователя глобально; создание заказа
  и донабор (рассылают уведомления всем подходящим исполнителям) — 5/мин.
- **Бан действует глобально**: забаненный пользователь получает 403 на все
  эндпоинты, кроме `GET /api/me`.

## Функционал

- **Категории работ**: погрузка / разгрузка / монтаж. Фильтр-чипы в ленте;
  исполнитель в настройках выбирает, о каких категориях получать уведомления
  (по умолчанию — обо всех).
- **«Сначала предложить избранным»**: чекбокс при создании заказа. Избранные
  исполнители работодателя получают уведомление сразу (с ⭐), остальные — через
  10 минут (фоновым джобом, переживает рестарт: волна хранится в БД).
- **Подтверждение выхода**: за час до начала смены принятым исполнителям приходит
  напоминание; кнопка «Подтвердить выход» — в карточке заказа. Работодатель видит
  статус подтверждения в списке откликов и может заранее открыть донабор.
- **Счётчик неявок**: при снятии исполнителя работодатель может отметить «не вышел
  на смену» — отметка видна в профиле рядом с рейтингом. Отзыв при снятии
  опционален. Отказаться от заказа может только работодатель.
- **Отзывы из трёх оценок**: пунктуальность / качество / адекватность; итоговый
  рейтинг — округлённое среднее. История отзывов с разбивкой и комментариями —
  на публичной странице профиля (тап по исполнителю или заказчику).
- **Автозакрытие просроченных заказов**: открытый заказ с прошедшей датой без
  принятых исполнителей отменяется (работодатель уведомляется); с принятыми —
  переводится «в работу». На следующий день после смены работодателю приходит
  напоминание завершить заказ и оценить исполнителей.
- **Поддержка**: кнопка в настройках — прямая ссылка на Telegram поддержки.
- **Оплата** производится работодателем напрямую исполнителю; сервис деньги
  не принимает и комиссию не берёт.

## Фоновый планировщик

`backend/src/jobs/scheduler.ts`, тик раз в минуту: вторая волна рассылки
«сначала избранным», автозакрытие просроченных, напоминание о завершении,
напоминание «подтвердите выход». Все задачи идемпотентны (защита от дублей
полями `broadcast_done`, `complete_reminder_sent_at`, `reminder_sent_at`) и
изолированы друг от друга по ошибкам. Таймзона процесса (`TZ`, по умолчанию
Europe/Moscow в compose) определяет «сегодня» для дат смен.

## Бизнес-правила (важные решения)

- **Новичок без отзывов** (`ratingCount = 0`) видит все заказы и может откликаться
  независимо от `minRating` заказа: порог — фильтр по репутации, а у новичка её ещё
  нет. Порог начинает действовать после первого отзыва.
- **Порог рейтинга проверяется и на сервере** при отклике — знание id заказа не
  обходит фильтр ленты.
- **Чёрный список действует в обе стороны**: заблокированные пары «работодатель ↔
  исполнитель» не видят заказы друг друга в ленте, не могут откликаться и не
  получают уведомления.
- **Отзывы при мультислоте**: участники завершённого заказа — работодатель и все
  принятые исполнители; работодатель оценивает каждого исполнителя, исполнители —
  работодателя (не друг друга).
- **Снять исполнителя** можно только с активного заказа (`open`/`in_progress`);
  снятие с завершённого ломало бы историю и статистику задним числом.

## Что исправлено относительно исходной версии

**Деплой.** Сгенерированы и закоммичены миграции (`backend/drizzle/`) — без них
Docker-образ не собирался (`COPY /app/drizzle` падал). Миграции применяются при
старте backend через `drizzle-orm` (drizzle-kit больше не нужен в прод-образе).
Добавлены `package-lock.json` во все три пакета; CI и Dockerfile переведены на
`npm ci`. Из деплой-скрипта убран `docker compose pull` (падал для build-сервисов
без registry). Исправлены две ошибки, из-за которых не проходили `typecheck`
backend (импорт ioredis под NodeNext) и сборка frontend (отсутствовал
`vite-env.d.ts`).

**Безопасность.** Убран захардкоженный Telegram ID администратора. Порты
PostgreSQL/Redis закрыты. Бан применяется глобально в auth-мидлваре. Порог
рейтинга и чёрный список проверяются на сервере при отклике. `GET /api/orders/:id`
больше не отдаёт `telegram_id` работодателя (для связи достаточно username).
Добавлен rate limiting. CORS корректно обрабатывает пустую переменную
(раньше `"".split(",")` давал `[""]` и блокировал все origin).

**Логика.** Отзывы работают для мультислотовых заказов (раньше участником считался
только первый принятый исполнитель — `.limit(1)`). Донабор и снятие исполнителя
выполняются в транзакции с `FOR UPDATE` (гонки с параллельным приёмом откликов).
Запрещено снятие исполнителя с завершённого заказа. `PATCH /api/me` с пустым телом
возвращает 400 вместо 500. Лента отвечает `{ items, total, page, limit }` — клиент
знает про следующую страницу («Показать ещё»).

**Уведомления и клиент.** У запросов к Telegram API появился таймаут. `apiFetch`
не шлёт `Content-Type: application/json` без тела — из-за этого падали с 400 все
POST без тела (отклик, завершение, отмена). Вне Telegram показывается понятное
сообщение вместо вечной «Загрузки…». `alert`/`confirm` заменены на нативные
`showAlert`/`showConfirm` Telegram (с фолбэком). Дата в ленте парсится как
локальная (фикс «Сегодня» в таймзонах западнее UTC). Обработаны ошибки выбора
роли и снятия исполнителя.

**Ранее исправленное** (сохранено): кэш ленты с ключом по userId, `auth_date` TTL
и constant-time сравнение хэша initData, фоновые батчи уведомлений, обработка
координаты 0, JSON-схемы валидации, числовой рейтинг `numeric(3,2)`, защита
отзывов уникальным индексом, устойчивость к ошибкам Redis.

## Эндпоинты

Базовые группы: `/api/me`, `/api/orders*`, `/api/responses/:id/accept`,
`/api/responses/:id/remove`, `/api/reviews`, `/api/favorites`, `/api/blacklist`,
`/api/employer/dashboard`, `/api/worker/stats`, `/api/admin/*`.
`GET /api/orders` возвращает `{ items, total, page, limit }` и принимает
`?category=`. Новое: `POST /api/responses/:id/confirm` (подтверждение выхода),
`GET /api/orders/:id/reviews/mine`, `myResponse` в деталях заказа,
`review`/`noShow` в теле снятия исполнителя, три составляющие оценки
в `POST /api/reviews`.
